## Using `pytraceview` safely from a host GUI

This advice was generated by GPT Codex as a result of its findings doing a second-source 
TraceLab+PyTraceView post-split code review and rework session.
This advice was not immediately deeply reviewed by the human developer, but seems a
correct conclusion to avoid some issues that were solved in the cross-contamination
and inappropriate private access fixes throughout.

### Core rule
Treat `TraceView` as the single source of truth for active plotting state.

Do not read hidden split-lane widgets while overlay is active.
Do not read overlay-only state while split mode is active.
Always branch on `trace_view.display_mode()` or use a `TraceView` helper that already does that.

### Preferred host-calling pattern
Use `TraceView` public methods for all host integrations:

- `display_mode()`
- `get_current_view_range()`
- `get_x_axis_item()`
- `overlay_plot_item()`
- `get_cursor_position()`
- `get_cursor_positions()`
- `refresh_trace_render(trace_name)`
- `refresh_all()`
- `grab_active_plot_area()`
- `set_range_bar_date_indicator(...)`
- `scroll_trace_list(...)`

Avoid reaching into private internals like `_lanes`, `_overlay_visuals`, `_overlay_widget`, `_range_bar`, `_mode`, or cached axis objects.

### Mode-parity rule
If a host action means “refresh one trace”, “read the active axis”, “update cursor readout”, or “export the visible view”:
implement it once, and let `TraceView` decide split vs overlay as late as possible.

Do not maintain separate host-side code paths unless the behavior is truly different.

### Cursor/value rule
Cursor values must follow the displayed primary segment, not just segment 0.

If a trace has overlapping segment time axes and `primary_segment` changes:
- the rendered curve must update
- the cursor readout must update immediately
- any label/value lookup must prefer the active primary segment first

After any trace-state change that affects displayed values, re-emit or refresh cursor values even if the cursor itself did not move.

### Axis/tick rule
Status-bar readouts must use the active axis only.

Examples:
- X/div in overlay must come from the overlay X axis, not a hidden split lane
- Y/div in overlay must be the one shared overlay Y axis for all visible traces
- Y/div in split may be per-lane

Never assume “if lanes exist, use a lane”; hidden lanes may still be alive after mode switches.

### Refresh rule
When model state changes, choose the smallest correct refresh:

- One trace changed visually: `refresh_trace_render(trace_name)`
- Many traces changed or global style changed: `refresh_all()`
- Theme/style change: restyle existing visuals in place, do not recreate traces unless structure changed

If the host shows derived UI like cursor panels or status bars, refresh that UI after non-cursor changes too.

### Export rule
“Visible export” must be based on the current visible X range and display-time arrays.

Do not assume:
- the first visible trace defines the export time basis
- every trace shares the same sample grid
- segmented traces should export full data just because they are segmented

Visible export should clip to `trace_view.get_current_view_range()` and then apply the export policy explicitly.

### Architecture guideline
A good public `TraceView` API should answer:
- What is the active mode?
- What is the active viewport?
- What is the active axis object?
- How do I refresh one trace correctly?
- How do I read cursor positions/values correctly?

If host code needs private widget access to answer one of those, add a public method instead.
